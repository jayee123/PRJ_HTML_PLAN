01

專題動機

日韓藥妝市場龐大，消費者卻面臨資訊分散、難以個人化篩選的困境。

痛點分析
🌐

資訊太分散

評測散落在多個不同平台，使用者需花大量時間自行比對，沒有統一的查詢入口。

🤔

難以個人化篩選

搜尋結果無法依照「乾性肌、預算 500 元」同時篩選，使用者只能靠運氣找到適合自己的產品。

📦

商品資訊不完整

日韓原文商品名稱、成分表、適用膚質標示不統一，台灣消費者需要自行翻譯與比對，門檻高。

核心痛點一句話： 消費者在購買日韓藥妝時，必須自己整合來自多個平台的資訊、再手動篩選符合自身條件的商品—— 這個過程耗時且容易做出不符合自身需求的購買決策。
02

目的

建立一個以資料驅動的日韓藥妝推薦平台，讓使用者不需要自己爬文比較，系統替他做完這件事。

1

聚合多平台評論，計算客觀好評率

將 PTT、Dcard、YouTube 的評論透過 AI 情緒分析模型自動判定每則評論的正負傾向，聚合多平台結果為每個商品計算出「社群好評率」，取代人工逐篇閱覽比對。

2

建立結構化商品資料庫（官方 API）

使用日本樂天 API 與韓國 Naver API 取得官方商品資料，確保商品名稱、圖片、成分、售價的準確性與合法性。

3

RAG 語意搜尋推薦

使用者輸入「乾性肌保濕，預算500」，系統透過 RAG 向量搜尋找到語意最相關的商品，不依賴關鍵字完全比對。

4

解決評論真偽問題

選擇 PTT（社群審查機制）與 Dcard（實名認證）作為主要評論來源，結構性地降低業配文影響，提高推薦可信度。

5

個人化篩選，降低購買錯誤率

依照膚質、品類、來源國、預算等條件多維度篩選，讓使用者找到真正適合自己的商品，減少衝動購買與退款。

04

進行方式

個人開發・CPU 筆電為主・8 週完成 MVP，採分階段驗證策略，先跑通再擴量。

4-1 開發時程（8 週）
第 1 週

申請 API + 環境建置

  • 申請 Rakuten Ichiba API Key（日本）
  • 申請 Naver Developers API Key（韓國）
  • 建立 MongoDB Atlas 免費叢集
  • 設計 products / reviews NoSQL Schema
  • 跑通 API 測試：各抓 10 個商品確認格式
第 2 週

商品資料批次擷取

  • Rakuten API：日系保養/防曬/底妝分類，目標 300 SKU
  • Naver API：韓系保養/防曬，目標 200 SKU
  • 建立品牌別名表（品牌中/日/韓/英對照）
  • rapidfuzz 做跨語言商品名稱去重
第 3 週

評論語料爬取

  • PTT MAKEUP 板：抓近 1 年，目標 3,000 篇
  • Dcard 美妝版：非官方 API 取 JSON，目標 2,000 篇
  • YouTube：yt-dlp 下載自動字幕（500 支）
  • rapidfuzz 評論關聯到對應商品 product_id
第 4 週

情緒分析 + 評分計算

  • HuggingFace 中文情緒模型跑全部評論
  • 計算每個商品：好評率 × log(提及數) 綜合分數
  • 更新 products.score / sentiment_pos_rate
  • 人工抽查 Top 20 商品分數合理性
第 5 週

RAG 向量化

  • sentence-transformers 建立商品向量索引（ChromaDB）
  • 批次跑評論 ↔ 商品配對（語意相似度）
  • 測試使用者查詢效果（3 個情境）
  • 調整相似度門檻，確保配對準確率
第 6 週

FastAPI 推薦系統

  • 實作 /products/recommend（篩選 + RAG 排序）
  • Upstash Redis 快取熱門查詢結果
  • 目標 API 回應 < 500ms
  • 本機壓力測試
第 7 週

前端整合 + 雲端部署

  • 前端 HTML 接真實 API，商品圖片用官方 API 圖片 URL
  • MongoDB Atlas 資料遷移（只存摘要欄位）
  • FastAPI 部署 Render.com
  • GitHub Pages 部署前端
第 8 週

P2 加分 + 簡報準備

  • GPT-4o 生成商品摘要（P2）
  • Whisper 補跑 300 支無字幕影片（Colab）
  • 整理 Demo 影片，準備老師攻擊問題標準回答
  • 最終壓力測試與效能調整
4-2 評審問題預演
商品資料完整嗎？你爬得到所有日韓商品嗎？最常問
答： 我們不是靠爬蟲，而是使用官方 API。 日本商品使用 Rakuten Ichiba API（日本最大電商，每日 3 萬次查詢免費）， 韓國商品使用 Naver Shopping API（韓國最大搜尋引擎，每日 2.5 萬次查詢）。 這兩個 API 的資料與消費者在這兩個平台上購物時所見完全一致，覆蓋市面上主流日韓藥妝品牌。
你的評論是真實用戶還是業配？怎麼保證？核心問題
答： 我們在設計上就選擇「結構性可信」的平台，而非試圖辨識業配文：
① PTT 美妝板：社群推文機制讓使用者自行審查，歷史上多次有業配文被肉搜公開，具公信力
② Dcard：大學生實名認證，每個帳號每天只能發一篇，刷評論成本極高
③ YouTube：設觀看數門檻（>1 萬），低觀看數業配較難滿足此條件
我們刻意不使用 Instagram，因其反爬措施強且業配辨識困難，這是有根據的設計決策。
RAG 是必要的嗎？直接用關鍵字搜尋不行嗎？技術深度
答： 有兩個具體原因：
① 評論配對：PTT 文章說「這款凝凍超補水」，關鍵字搜尋找不到「肌ラボ極潤」， 向量搜尋可以跨語言、跨表達方式做語意配對，準確率高出 30%+
② 使用者查詢：使用者說「不黏膩的保濕」，關鍵字搜尋只能找含這個詞的結果， RAG 可以找到描述為「水感質地」「清爽不油」的商品，語意更豐富
一個人做得完嗎？8 週夠嗎？可行性
答： 已在筆電本機完成 Demo 驗證（20 個商品 + 50 篇評論跑通完整流程）。 API 申請、爬蟲、情緒分析、RAG 向量化、FastAPI、部署都已有對應的程式碼框架。 8 週時程含 Buffer，P1 核心功能在第 7 週完成，第 8 週為加分項目。 複雜度最高的 Whisper 轉文字已規劃用 Google Colab 免費 GPU 分批處理，不依賴本機算力。
