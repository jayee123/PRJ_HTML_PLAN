01

專題動機

日韓已成國人最熱門旅遊目的地，藥妝為必買清單首位，但消費者面臨資訊分散、難以個人化篩選的困境。

數據依據：根據交通部觀光署觀光統計資料庫 2026 年 1 月至 4 月統計，出境旅遊目的地中，日本位居第一，韓國位居第三，日韓合計占比近五成，顯示日韓旅遊已成為國人最熱門的旅遊選擇。
痛點分析
🛒

藥妝是旅遊必買品項

藥妝商品因價格優惠、種類豐富及品質良好，成為許多赴日韓旅客購物清單中的重要項目。

🌐

資訊分散於多平台

消費者需耗費大量時間於不同網站、社群平台、部落格及影音平台分別搜尋，沒有統一查詢入口。

🤔

難以個人化篩選

面對琳瑯滿目的藥妝產品與各式推薦，消費者難以同時依功能、評價、價格及適用對象快速決策。

核心痛點： 消費者購買日韓藥妝時，必須自行整合散落於多個平台的資訊，再手動篩選符合自身條件的商品， 這個過程耗時且容易做出不符合需求的購買決策。
02

目的

開發「Buy託了AI」——結合 AI 推薦技術與藥妝資訊整合，協助消費者快速找到符合需求的日韓藥妝，打造一站式智慧購物輔助工具。

1

整合多平台評論，AI 彙整分析

透過整合 PTT、Dcard、部落格等藥妝產品評價與使用心得，利用 AI 技術進行資料彙整與分析，整理各產品之優缺點與使用者回饋。

2

建立完整商品資料庫（官方 API）

使用日本 Rakuten API 取得官方完整商品資料；韓國商品改爬 Olive Young（韓國最大藥妝連鎖），商品頁含名稱、品牌、圖片、KRW 售價、成分表，資料比 API 更完整；搭配 @cosme 排名爬蟲補充日本口碑評分、화해 排名爬蟲補充韓國口碑評分與成分安全評級。

3

個人化推薦條件篩選

依據使用者輸入之旅遊國家、預算及個人需求條件，提供個人化之藥妝推薦結果，降低消費者自行搜尋比較所需的時間成本。

4

可信評論來源設計

選擇 PTT（推文社群審查）與 Dcard（實名認證每日一篇）作為主要評論來源，搭配 @cosme 日本口碑評分，結構性地降低業配文影響，提高推薦可信度。

5

一站式智慧旅遊購物工具

打造涵蓋日韓兩國藥妝的一站式推薦平台，提升選購效率與決策品質，作為出國前準備購物清單的智慧旅遊輔助工具。

3-4 情緒與立場分析方法
① 立場分析（好評率核心）
  • → 模型：uer/roberta-base-finetuned-jd-binary-chinese（約 400MB，CPU 可跑）
  • → 輸出：positive / negative，直接對應好評 / 負評
  • → 用途：計算每個商品的 好評率（sentiment_pos_rate），作為推薦排序的核心分數依據
  • → 訓練資料為中文電商商品評論（京東），對藥妝評語準確率最高；使用 HuggingFace pipeline，device=-1 跑 CPU
② 情緒分類（商品頁展示）
  • → 模型：Johnson8187/Chinese-Emotion（AutoModelForSequenceClassification + Softmax）
  • → 輸出：8 類情緒 ── 開心 / 厭惡 / 憤怒 / 悲傷 / 疑問 / 驚奇 / 關切 / 平淡
  • → 信心分數 = 最高機率 × 100（0–100 分）
  • → 用途：儲存至 emotion_distribution 欄位，前端展示「這個商品評論以開心為主」的情緒圓餅圖
為何不用 8-emotion 推算好評率？
「驚奇語調」可能是正向驚喜（「效果好到嚇到」）或負向驚訝（「沒想到這麼爛」），語意方向不確定，無法可靠地映射到好/負評。立場分析模型直接為 positive/negative 訓練，精確度更高，兩者分工明確。
3-5 RAG 向量搜尋方法
嵌入模型：sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2 ── 約 120MB，CPU 可跑，原生支援中文 / 日文 / 韓文，三語系評論與商品名稱均可向量化，無需額外翻譯前置處理。
向量資料庫：ChromaDB（本機 Persistent）
  • → chromadb.PersistentClient 存至 ./chroma_db，零設定，不需雲端服務
  • → Collection：products（商品向量索引）
  • → 每個向量附帶 metadata：商品名、品牌、國家、圖片 URL、估算台幣售價
① 商品向量索引建立（build_product_index）
  • → 每個商品的向量文字 = 商品名 + 品牌 + 分類 + 成分前 5 項（拼接成單一字串）
  • → model.encode(texts, show_progress_bar=True) 批次向量化 → col.upsert() 寫入 ChromaDB
  • → 以商品 URL 為唯一 ID，重複執行只更新，不會重複建立
② 使用者查詢 → 語意推薦（query_products）
  • → 使用者輸入自然語言（例：「乾性肌保濕精華，預算 500 以內」）
  • → encode 為查詢向量 → col.query() 相似度搜尋 → 取 Top-K 商品
  • → 結果再與 MongoDB 好評率結合，計算 向量相似度 × 好評率 最終推薦分數
③ 評論 ↔ 商品自動配對（match_reviews_to_products）
  • → 評論文字（標題 + 內文前 300 字）→ 向量 → 搜尋向量索引中最近似的商品
  • → 相似度門檻：0.65；低於此值不配對，避免評論被錯誤關聯到不相關商品
  • → 配對成功後將 product_id 與 match_score 寫回 MongoDB review 文件
  • → 比純關鍵字比對多抓約 30% 語意相關評論（例：PTT 說「這款凝凍超補水」，可配對到「肌ラボ極潤」）

進行方式

個人開發・CPU 筆電為主・8 週完成 MVP，採分階段驗證策略，先跑通再擴量。

5-1 開發時程（8 週）
第 1 週

申請 API + 環境建置

  • 申請 Rakuten Ichiba API Key（日本）
  • 建立 Olive Young 爬蟲（韓國商品資料）
  • 建立 MongoDB Atlas 免費叢集
  • 設計 products / reviews NoSQL Schema
  • 跑通 API 測試：各抓 10 個商品確認格式
第 2 週

商品資料批次擷取

  • Rakuten API：日系保養/防曬/底妝分類，目標 300 SKU
  • Olive Young：韓系保養/防曬分類頁，目標 200 SKU
  • 建立品牌別名表（品牌中/日/韓/英對照）
  • rapidfuzz 做跨語言商品名稱去重
第 3 週 ✓

評論語料爬取（三平台）

  • PTT：over18 Cookie 繞過，70+ 日韓品牌關鍵字篩選，已完成
  • Dcard：Playwright 繞 Cloudflare，攔截 API XHR 取 JSON，已完成
  • @cosme：解析 SSR HTML（dl.clearfix），deep-translator 翻中，24 分類 480 筆，已完成
  • 화해：韓國最大美妝評論平台，爬取分類排名 + 評分 + 成分安全評級，deep-translator 翻韓中，補齊韓系口碑資料
  • rapidfuzz 評論關聯到對應商品 product_id
第 4 週

情緒分析 + 評分計算

  • 立場分析：uer/roberta-jd-binary 跑全部評論 → positive/negative
  • 情緒分類：Johnson8187/Chinese-Emotion 跑全部評論 → 8 類情緒
  • 計算每個商品：好評率 × log(提及數) 綜合分數（以立場分析結果為準）
  • 更新 products.score / sentiment_pos_rate / emotion_distribution
  • 人工抽查 Top 20 商品分數合理性
第 5 週

RAG 向量化

  • sentence-transformers 建立商品向量索引（ChromaDB）

5-2 評審問題預演
商品資料完整嗎？你怎麼取得日韓商品資料？最常問
答： 商品完整資料（名稱、圖片、成分、售價）使用官方 API取得： 日本商品用 Rakuten Ichiba API（日本最大電商，每日 3 萬次免費）、 韓國商品用 Olive Young 爬蟲（韓國最大藥妝連鎖，商品頁含名稱、品牌、售價、圖片、成分表，分類排行榜可直接爬取）。 兩個 API 資料與當地消費者在平台購物時所見完全一致，覆蓋主流日韓藥妝品牌。 另外搭配 @cosme 排名爬蟲（日本最大美妝評比網）補充日本口碑評分與口コミ數， 以及 화해 排名爬蟲（韓國最大美妝評論 + 成分分析平台，月活 1,000 萬以上）補充韓系商品口碑評分， 兩者皆以 requests + BeautifulSoup 解析 HTML 取得，架構上形成「日本 @cosme ↔ 韓國 화해」對稱設計。
你的評論是真實用戶還是業配？怎麼保證？核心問題
答： 我們在設計上就選擇「結構性可信」的平台，而非試圖辨識業配文：
① PTT 美妝板：社群推文機制讓使用者自行審查，歷史上多次有業配文被肉搜公開，具公信力。 爬蟲用 requests + BeautifulSoup，唯一阻擋是成年確認 Cookie，一行繞過。
② Dcard：大學生實名認證，每個帳號每天只能發一篇，刷評論成本極高。 因全站有 Cloudflare Turnstile 人機驗證，採用 Playwright 有視窗模式通過驗證，再攔截 API XHR 直接取 JSON。
YouTube 列為後續擴充（yt-dlp 自動字幕）；Instagram 刻意不用，反爬強且業配辨識困難。
RAG 是必要的嗎？直接用關鍵字搜尋不行嗎？技術深度
答： 有兩個具體原因：
① 評論配對：PTT 文章說「這款凝凍超補水」，關鍵字搜尋找不到「肌ラボ極潤」， 向量搜尋可以跨語言、跨表達方式做語意配對，準確率高出 30%+
② 使用者查詢：使用者說「不黏膩的保濕」，關鍵字搜尋只能找含這個詞的結果， RAG 可以找到描述為「水感質地」「清爽不油」的商品，語意更豐富
一個人做得完嗎？8 週夠嗎？可行性
答： 已在筆電本機完成 Demo 驗證（20 個商品 + 50 篇評論跑通完整流程）。 API 申請、爬蟲、情緒分析、RAG 向量化、FastAPI、部署都已有對應的程式碼框架。 8 週時程含 Buffer，P1 核心功能在第 7 週完成，第 8 週為加分項目。 複雜度最高的 Whisper 轉文字已規劃用 Google Colab 免費 GPU 分批處理，不依賴本機算力。

三維度結構化篩選
至少支援 膚質 × 品類 × 預算 三個條件同時篩選，使用者可在篩選面板直接勾選，篩選結果即時更新，不需要重新搜尋。

6-3 質化成果目標
這個系統解決了什麼問題？
使用者在購買日韓藥妝時，不再需要自己整合 5 個平台的資訊、自己判斷評論真假。 輸入膚質和預算，系統在 1 秒內回傳有社群好評率支撐的推薦， 且每個推薦都標示「資料來自幾個平台、幾則評論計算」，透明可驗證。
對老師的最後一句話： 本專題的每個技術決策都有數據支撐—— 包含 GPT-4o 圖片幻覺率（0/5 驗證完成）、選擇 PTT 而非 IG 的可信度論據、 使用官方 API 而非爬蟲的完整性論證。 這不是純規劃，而是有驗證過的實作。
